# claudesync v0.2 — Sync button + universal repo support

**Дата:** 2026-05-06
**Статус:** Approved
**Goal:** Replace per-OS "Обновить" buttons with a single **Sync** button. Add Repo URL field so claudesync can clone-or-pull any compatible repo and install its rules to a user-configured target folder. Decouples claudesync from a hardcoded `~/.claude/` path — works for any AI-tool config repo following the convention.

## Convention for compatible repos

A repo is "claudesync-compatible" if:
- Has `install.sh` (macOS/Linux) or `install.ps1` (Windows) in root
- The install script honors `RULES_TARGET` env var, falling back to `$HOME/.claude` (Unix) or `$HOME\.claude` (Win) for backwards compat

`DuchitskyDA/ai` will be updated alongside this release to match.

## Settings — three fields

1. **Repo URL** — `https://github.com/...` or `git@github.com:...`. Validated by simple regex.
2. **Local repo path** — where to clone. Browse picker. No default — user picks.
3. **Rules target folder** — where install script links files into. Browse picker. Default suggestion: `~/.claude`.

All three persisted in `userData/config.json` as `AppConfig`. Backwards compat: existing `repoPath` field remains as Local repo path; new `repoUrl` and `rulesTarget` are added.

## Sync flow

Single button **Sync** (label "Sync now"):

1. Validate: all three fields filled, URL matches regex, paths are absolute strings.
2. If `<localRepoPath>/.git` does NOT exist → `git clone <repoUrl> <localRepoPath>`
3. Else → `git -C <localRepoPath> pull`
4. If exit ≠ 0 → fail
5. Detect platform script: `install.sh` (darwin/linux) or `install.ps1` (win32). Error if missing.
6. Run install with `RULES_TARGET=<rulesTarget>` env var:
   - macOS/Linux: `bash install.sh` with env
   - Windows: `powershell -ExecutionPolicy Bypass -File install.ps1` with env
7. Stream stdout/stderr to log as before. Final ✓ DONE / ✗ FAILED.

## UI changes

- Two `UpdateButton` (macos + windows) → one `<SyncButton>` always enabled when config is complete and not running.
- Disabled conditions: any of three fields empty, OR running.
- Tooltip: "Configure repo URL, local path, and rules folder in Settings".
- Header repo chip stays — shows `localRepoPath`.
- Settings modal grows: 3 labeled inputs + 2 Browse buttons. Cancel / Save buttons unchanged.

## Validation rules

- **Repo URL**: matches `/^(https?:\/\/|git@)[\w\-.@:/~]+\.git?$/i` (lenient — accepts `.git` ending or not)
- **Local repo path**: must be absolute (`path.isAbsolute`). If exists → must be empty OR be a git repo whose `origin` matches `repoUrl` (warn, don't block).
- **Rules target folder**: must be absolute. Will be created if absent during install.

Validation runs in `setConfig` IPC handler, returns `{ ok: false, error: ... }` on fail.

## Companion change in `DuchitskyDA/ai`

Update `install.sh`:
```diff
-CLAUDE_DIR="$HOME/.claude"
+CLAUDE_DIR="${RULES_TARGET:-$HOME/.claude}"
```

Update `install.ps1`:
```diff
-$ClaudeDir = Join-Path $HOME '.claude'
+$ClaudeDir = if ($env:RULES_TARGET) { $env:RULES_TARGET } else { Join-Path $HOME '.claude' }
```

This is a separate commit/push to `DuchitskyDA/ai` repo.

## Tests

- `runner.ts`: no changes needed — already supports env via spawn options. Add overload `runCommand(cmd, args, { cwd, env?, onLine })`.
- `ipc.ts`: new `runSyncHandler(deps)` tested for: missing URL, missing local path, missing rules target, fresh clone (no .git), pull (existing .git), missing install script, bad git pull exit, success.
- `config.ts`: extend `AppConfig` and `validateRepoPath` (renamed `validateLocalRepo` since rules now apply to local clone too) to accept `repoUrl: string | null`, `rulesTarget: string | null`. Tests for empty URL, malformed URL, etc.
- Renderer: no unit tests (minimal logic).

## Versioning

Tag `v0.2.0` after Sync ships. CI publishes new release.

## Out of scope

- Auto-update of claudesync itself (still v3+)
- Multiple repo profiles (one repo per install for now)
- SSH key management UI (user's git client handles auth)
- Inline display of remote `origin` mismatch (warn only via setConfig validation, not blocking UI)
