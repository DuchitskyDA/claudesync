# claudesync v0.5 — i18n + Conflict Resolution

**Дата:** 2026-05-08
**Статус:** Draft (local-only, ожидаются правки; релиз отложен)
**Goal:** Добавить интернационализацию (RU + EN + auto-detect системной локали) и интерактивный резолвер merge-конфликтов при Push (read-only three-way diff + per-file mine/theirs + escape hatch в редактор), который замещает текущий `rebase --abort` + «resolve manually».

## Контекст

После v0.4 (см. [2026-05-08-v0.4-bidirectional-sync.md](2026-05-08-v0.4-bidirectional-sync.md)):

- Push выполняет `git pull --rebase --autostash` перед push'ем.
- При конфликте `runPush` делает `git rebase --abort` и возвращает `failResult('Merge conflict during rebase. Resolve manually in <repoPath>...')`.
- Three-way merge UI был явно отнесён к v0.5.
- Все строки UI и main-process сообщения захардкожены английским в ~17 компонентах renderer (~1700 строк TSX) и в нескольких main-файлах (`push.ts`, `init-wizard.ts` и т.д.).

v0.5 решает обе задачи в рамках одного релизного цикла.

### Ключевые архитектурные принципы

1. **main — i18n-blind.** main никогда не знает о текущем языке. Все строки, попадающие в renderer, передаются как `LocalizedMessage = { key, params?, fallback? }`. Renderer переводит. Сырой git stdout/stderr остаётся в логе как есть.

2. **Конфликт не отбрасывается автоматически.** При `classifyPullError === 'conflict'` НЕ делаем `rebase --abort`. Оставляем rebase в paused-состоянии и эмитим событие со списком unmerged-файлов. Пользователь сам решает: разрулить или Abort.

3. **Recovery после рестарта.** На старте app `useAppState` вызывает `conflictGetState()`. Если `inProgress === true` — показывается баннер «Unresolved push from previous session», открывается `ConflictModal`. Это бесплатное следствие принципа №2.

4. **Локаль в `AppConfig.locale: 'en' | 'ru' | null`.** `null` = follow-system (`app.getLocale()`). Любой явный выбор в Settings перекрывает.

## Conflict flow

### Терминологическая ловушка `git rebase`

Внутри `git pull --rebase` git инвертирует `--ours` / `--theirs`:

| User mental model     | git rebase term | git index stage | Команда                       |
|-----------------------|-----------------|-----------------|-------------------------------|
| **Mine** (это машина) | `--theirs`      | stage `:3:`     | `git checkout --theirs <p>`   |
| **Remote** (GitHub)   | `--ours`        | stage `:2:`     | `git checkout --ours <p>`     |
| **Base** (общий пред) | —               | stage `:1:`     | `git show :1:<p>`             |

В `conflict.ts` константы:

```ts
const STAGE_BASE = 1
const STAGE_REMOTE = 2  // git's --ours during rebase
const STAGE_MINE = 3    // git's --theirs during rebase
```

### State machine push

```
runPush
  → export → check status clean
  → git pull --rebase
       ├─ exit 0 ────────→ commit → push → done
       ├─ network/auth ──→ retry once → fail (toast)
       └─ conflict ──────→ emitStep({ step: 'pull', status: 'failed', kind: 'conflict' })
                            return { ok: false, kind: 'conflict' }
                            ⚠ rebase остаётся paused, НЕ abort'им
```

Renderer при `result.kind === 'conflict'` открывает `ConflictModal` (вместо toast).

### IPC API

```ts
// shared/api.ts

export type ConflictFileStatus =
  | 'unresolved'
  | 'resolved-mine'
  | 'resolved-remote'
  | 'resolved-manual'

export type ConflictFile = {
  path: string
  status: ConflictFileStatus
  binary: boolean       // null-byte в первых 8 KB либо git --numstat возвращает '-'
}

export type ConflictState = {
  inProgress: boolean   // false = нет paused rebase
  files: ConflictFile[]
}

export type ConflictSide = 'base' | 'remote' | 'mine'

interface AppApi {
  // ... existing
  // i18n
  getSystemLocale(): Promise<string>                         // app.getLocale() raw, e.g. "ru-RU"
  // conflict
  conflictGetState(): Promise<ConflictState>
  conflictGetFile(path: string, side: ConflictSide): Promise<{ text: string | null; binary: boolean }>
  conflictResolveFile(
    path: string,
    choice: 'mine' | 'remote' | 'manual',
  ): Promise<{ ok: boolean; error?: LocalizedMessage }>
  conflictOpenInEditor(path: string): Promise<void>          // shell.openPath
  conflictContinue(): Promise<RunResult>                     // rebase --continue → push
  conflictAbort(): Promise<void>                             // rebase --abort
}
```

### Per-file resolve (что именно делает main)

```ts
// choice='mine'
git -C <repo> checkout --theirs -- <path>   // stage :3 → working tree
git -C <repo> add -- <path>

// choice='remote'
git -C <repo> checkout --ours -- <path>
git -C <repo> add -- <path>

// choice='manual' (после того как пользователь сохранил файл из редактора)
// Проверка: оставшихся <<<<<<< / ======= / >>>>>>> в файле нет — иначе error
git -C <repo> add -- <path>
```

### `conflictContinue`

1. Pre-check: `state.files.every(f => f.status !== 'unresolved')`. Если нет — UI кнопка disabled, до сюда не дойдёт.
2. `git -C <repo> -c core.editor=true rebase --continue` (`core.editor=true` чтобы git не открыл редактор для commit-msg).
3. Если `rebase --continue` снова даёт конфликт (следующий коммит в стэке) — повторно эмитим событие, модал перерисовывается с новым списком.
4. Если rebase завершён — продолжаем оригинальный flow `runPush`: `commit -A` (no-op) → `push origin main`.

### `conflictAbort`

`git -C <repo> rebase --abort` → `state.inProgress = false` → модал закрывается.

### Recovery на cold start

`useAppState` при инициализации зовёт `conflictGetState()`. Если `inProgress === true` — persistent-баннер «Unresolved push from previous session» + кнопка «Resolve» открывает `ConflictModal`. До резолва Pull/Push кнопки disabled (`isRunning`-семантика).

### `ThreeWayDiff` (read-only)

Три колонки сырого текста (`<pre>`), синхронный скролл. Подсветка построчная:

- строка отличается от `base` → жёлтый фон;
- строки нет в текущем stage (delete) → серый зачёркнутый.

Без char-level diff, без библиотек. Если `binary === true` или файл больше 500 KB — три колонки заменяются placeholder'ом «File not previewable — open in editor». Если в `:1:` пусто (add/add — оба добавили файл) — колонка base пустая, остальные две сравниваются между собой.

### Формат словаря

Плоские dotted keys; для плюрализации значение становится объектом с CLDR-категориями.

```json
{
  "header.signIn": "Sign in",
  "header.signOut": "Sign out",
  "sync.noRepo": "No repo configured.",
  "push.error.network": "Network error talking to GitHub. Check VPN/proxy and click Push to retry.{{tail}}",
  "push.error.conflict": "Merge conflict — please resolve.",
  "conflict.unresolved": {
    "one":   "{{count}} unresolved file",
    "few":   "{{count}} unresolved files",
    "many":  "{{count}} unresolved files",
    "other": "{{count}} unresolved files"
  }
}
```

`ru.json` — те же ключи; для плюральных значений все четыре CLDR-категории RU (`one`, `few`, `many`, `other`).

### `t()` алгоритм

```ts
function t(key, params) {
  let raw = lookup(dict[currentLocale], key)
  if (raw == null && currentLocale !== 'en') raw = lookup(dict.en, key)   // fallback EN
  if (raw == null) {
    if (process.env.NODE_ENV !== 'production') console.warn(`i18n miss: ${key}`)
    return key                                                              // last resort
  }
  if (typeof raw === 'object') {
    const cat = new Intl.PluralRules(currentLocale).select(Number(params?.count ?? 0))
    raw = raw[cat] ?? raw.other ?? Object.values(raw)[0]
  }
  return raw.replace(/\{\{(\w+)\}\}/g, (_, k) => String(params?.[k] ?? ''))
}
```

### Инициализация локали

`LocaleProvider` mount:

1. `cfg = await window.api.getConfig()` — `cfg.locale: 'en' | 'ru' | null`.
2. Если `cfg.locale` непустой — берём.
3. Иначе IPC `getSystemLocale()` → main `app.getLocale()` (например `ru-RU`) → нормализация (`'ru-*' → 'ru'`, остальное → `'en'`).
4. Хранится в React state, меняется через `setLocale(null | 'en' | 'ru')` — без перезагрузки app.

### Мост main → renderer для текстов

```ts
// shared/api.ts

export type LocalizedMessage = {
  key: string
  params?: Record<string, string | number>
  /** raw english fallback — used if key missing OR if main can't form key (e.g. unparsed git stderr tail) */
  fallback?: string
}

export type RunResult = {
  ok: boolean
  exitCode: number
  error?: LocalizedMessage
  kind?: 'conflict'
}

export type PushStepEvent = {
  step: PushStep
  status: StepStatus
  message?: LocalizedMessage
}
```

Renderer-helper:

```ts
function tMessage(t: TFn, m: LocalizedMessage | undefined): string {
  if (!m) return ''
  const translated = t(m.key, m.params)
  return translated === m.key && m.fallback ? m.fallback : translated
}
```

### Миграция call-sites в main

Точечно по файлам:

- `push.ts::pullErrorMessage` — возвращает `LocalizedMessage` вместо строки.
- `push.ts::failResult` — принимает `LocalizedMessage`.
- `init-wizard.ts` — статусы шагов (`create-repo` / `clone` / `generate` / `commit` / `push`) переходят на ключи.
- Прочие константы `'Sync not configured'`, `'Not signed in to GitHub'`, `'commit failed'`, `'Nothing to push…'` — каждое получает ключ `push.error.*` / `init.error.*`.

Сырой `tail` от git stderr подставляется в `fallback` или `params.tail` — это уже английский git output, переводу не подлежит.

### Что НЕ переводится

- Сырой git output в `LogConsole` (stdout/stderr внешнего процесса).
- Имена файлов, paths, repo URLs.
- Названия шагов как internal id (`'export'`, `'pull'`, `'commit'`, `'push'`); их человекочитаемые лейблы в UI берутся через `t('step.export')` и т.д.
- Технические идентификаторы конфликта (`UU`, `:1:`, `--theirs` в логах).

### Manual E2E (расширение существующего из v0.4)

Существующие сценарии 1–8 из v0.4-spec повторяем как regression. Дополнительные:

9. **Conflict resolution — same-line edit:**
   - На двух машинах изменить одну строку в `~/.claude/CLAUDE.md`.
   - Push с machine A → ok.
   - Push с machine B → ConflictModal со списком `global/CLAUDE.md`.
   - Three-way diff показывает три колонки.
   - Выбор «Mine» → файл помечается resolved-mine, кнопка «Continue» enable.
   - Continue → push успешен, на GitHub именно версия с machine B.

10. **Conflict — manual editor path:**
    - Тот же сценарий, но «Open in editor», правим вручную, удаляем маркеры, сохраняем.
    - Mark resolved → файл resolved-manual.
    - Continue → push успешен.

11. **Conflict — multiple files:**
    - Три файла одновременно конфликтуют (`CLAUDE.md`, `settings.json`, один skill).
    - Резолв per-file разными способами (mine / remote / manual). Continue → push.

12. **Conflict — Abort:**
    - Конфликт → Abort → no rebase paused → `git status` чистый → можно делать новый Pull.

13. **Recovery после рестарта:**
    - Спровоцировать конфликт → закрыть app в середине резолва → открыть заново.
    - Banner «Unresolved push from previous session» виден; модал открывается с тем же state.

14. **i18n — language switch live:**
    - Settings → System → видим текущий системный язык.
    - Переключение на Русский → весь UI без рестарта.
    - Перезапуск app → язык сохранился.
    - Settings → English → обратно.

15. **i18n — error messages:**
    - Включить RU. Спровоцировать `network` ошибку (отрубить сеть) → toast/step-message по-русски.
    - То же для `auth` (revoke токена) и `conflict`.
    - Сырой git stderr в логе — на английском (это ожидаемо).

16. **i18n — RU plurals:**
    - В ConflictModal проверить, что `1 файл / 2 файла / 5 файлов / 21 файл / 22 файла` отображаются грамматически верно.

- Скриншоты UI / визуальный snapshot — RU и EN визуально проверяются вручную.

## Порядок работ — два последовательных плана

i18n трогает каждый компонент и каждый main-файл; conflict-резолвер — точечно. Это два независимых куска, ставить всё в один PR = огромный диф, в котором невозможно ревьюить.

| # | Спринт                                | Что включает                                                                                                                                                                            | Размер диффа    |
|---|----------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------|
| 1 | **i18n infrastructure + migration**    | `i18n/index.tsx`, словари EN/RU, миграция всех 17 компонентов, `LocalizedMessage`-payload в main, settings UI «Language», тесты на словари + `t()`                                       | ~25–30 файлов   |
| 2 | **Conflict resolver**                  | `conflict.ts` (main), `ConflictModal`, `ConflictFileList`, `ThreeWayDiff`, изменения в `push.ts` (no auto-abort + emit `kind: 'conflict'`), recovery-banner в `App.tsx`, новые ключи словарей, тесты | ~10–12 файлов   |

Conflict-фича сразу пишется i18n-native: никакого «потом переведём».

Каждый план будет описан отдельно (writing-plans skill) после ревью этого spec'а.

## Риски и mitigations

| Риск                                                                     | Mitigation                                                                                                                                                                                                                          |
|--------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Пропуск строк при i18n-миграции — на UI висит хардкод EN.                | Static-test `dictionaries.test.ts` ловит расхождения EN↔RU. Dev-console warn при `t()` miss. Один проход grep по `>[^<{]+<` в `*.tsx` после миграции (manual QA шаг).                                                                |
| Crash app в середине conflict-резолва → rebase paused.                   | Recovery-banner на cold start (детектит `.git/rebase-merge/`). Не теряем state — просто перерисовываем модал.                                                                                                                       |
| Пользователь руками (через CLI) ломает rebase-state между сессиями.      | Принимаем. UI просто покажет «no rebase in progress» — Abort-кнопка станет no-op, модал закроется. Документируем в README.                                                                                                          |
| Three-way diff на больших файлах → лагает, нет folding.                  | Принимаем для v0.5. Если файл > 500 KB — fallback на «File too large — open in editor». Char-level diff и folding — v0.6+.                                                                                                          |
| Binary-detection ложно-положителен (UTF-16 BOM и т.п.) → диф недоступен. | Стандартная эвристика «null-byte в первых 8 KB» + git's own `is_binary` (`git diff --numstat` returns `-` для бинарников) как secondary check. Workaround для пользователя — Open in editor.                                         |
| RU plural-формы заполнены неверно (англоговорящие контрибьюторы).        | `dictionaries.test.ts` требует все четыре категории для RU; контент валидирует maintainer.                                                                                                                                          |
| Live-смена локали не перерисует все компоненты.                          | `useT()` подписан на context — все consumers рендерятся при смене. Ключевой инвариант: в коде нет кеша строк вне рендеров.                                                                                                          |

## Out of scope (deferred → v0.6+)

- ❌ Char-level diff внутри строки в `ThreeWayDiff` (`diff` lib или ручная реализация).
- ❌ Inline-резолв (редактируемый merge-editor типа monaco) — был вариант C при выборе скоупа конфликт-резолвера, отвергнут.
- ❌ Folding неконфликтующих диапазонов в three-way diff.
- ❌ Syntax highlighting в three-way diff.
- ❌ Conflict-резолвер для **bare merge** (не rebase) — v0.5 всегда работает с `pull --rebase`.
- ❌ Auto-resolve правила («всегда mine для memory/», «всегда remote для settings.json»).
- ❌ Третий язык (de/fr/zh/...). Архитектура готова — добавление сводится к новому `xx.json` + ветке в normalize-функции системной локали.
- ❌ Перевод сырого git stdout/stderr.
- ❌ Локализация дат/чисел (`Intl.DateTimeFormat`, `Intl.NumberFormat`) — в v0.5 текстов с датами/числами почти нет.
